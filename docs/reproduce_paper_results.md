# Reproducing the Paper Results

This document maps the figures and the benchmark experiments in the paper to
the scripts that produce them. Set up the environment first (see
`system_setup.md`).

## Figures

Figure 1 is a schematic pipeline diagram; it is a static asset, finalized
manually, and is not regenerated by code. Figures 2-8 and the appendix figure
are produced by `plot_paper_figures.py`, which saves a PNG and a PDF for each
figure through the project plotting module.

To regenerate all code-generated figures:

```
python plot_paper_figures.py
```

| Figure | Content                                         | Generating function          |
|--------|-------------------------------------------------|------------------------------|
| Fig. 1 | Schematic pipeline overview (static asset)      | not code-generated           |
| Fig. 2 | Cart-Pole pass-level summary                    | `plot_fig2_gate3_pass_level` |
| Fig. 3 | Cart-Pole D-optimal vs. random                  | `plot_fig3_cp_dopt_random`   |
| Fig. 4 | AEK conditioning history                        | `plot_fig4_aek_history`      |
| Fig. 5 | AEK coverage diagnostic                         | `plot_fig5_coverage`         |
| Fig. 6 | Applicability Condition cross-system test       | `plot_fig6_applicability`    |
| Fig. 7 | Silverbox random-selection results              | `plot_fig7_silverbox`        |
| Fig. 8 | Four-system verdict synthesis                   | `plot_fig8_verdict`          |
| Fig. B.1 | Lynx-Hare appendix figure                     | `plot_figA2_lynxhare`        |

## Benchmark experiments

The scripts below regenerate the run outputs that the figures and tables
summarize. Each writes to `results/` following the project path convention. Run
a script with `python <script>.py`; pass `--help` to see any available
options.

Some script and function names retain legacy internal identifiers for
reproducibility; the descriptions in this document use the terminology of the
manuscript.

| Script                      | Experiment                                      |
|-----------------------------|-------------------------------------------------|
| `run_gate3_v2.py`           | Cart-Pole baseline and augmentation evaluation  |
| `run_gate4_ablation.py`     | Cart-Pole D-optimal vs. random ablation         |
| `run_aek3_baseline.py`      | AEK baseline evaluation                         |
| `run_aek4_ablation.py`      | AEK ablation and conditioning diagnostics       |
| `run_aek4c_random.py`       | AEK reparameterized, random selection           |
| `run_aek4c_dopt.py`         | AEK reparameterized, D-optimal selection        |
| `run_aek4c_dopt_dither.py`  | AEK D-optimal + dither (exploratory)            |
| `run_aek4c_dither_random.py`| AEK coverage-recovered (dither-plus), random    |
| `run_lorenz_gate.py`        | Lorenz-63 augmentation analysis                 |
| `run_lynxhare_gate.py`      | Lynx-Hare appendix analysis                     |
| `cp_rebaseline_savgol.py`   | Cart-Pole Savitzky-Golay rebaseline             |
| `fix_aek_savgol_clean.py`   | AEK Savitzky-Golay consistency patch            |

## Pool integrity check

Each benchmark draws from a fixed candidate pool. After regenerating a pool,
recompute its SHA fingerprint and compare it against `pool_sha_registry.md`. A
matching fingerprint confirms the pool is identical to the one used in the
paper; the registered fingerprints are immutable.

## Notes

- All derivatives, in both the pools and the training sets, are computed with
  the Savitzky-Golay filter. The repository contains no analytic-derivative
  path.
- The AEK trajectories are noise-free; no artificial measurement noise is added.
- Result tables in the paper are aggregated from the `metrics.json` files
  written under `results/`.
